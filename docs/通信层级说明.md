# 通信层级说明

## 1. 目的

本文用于整理当前项目中“通信是如何分层理解”的思路，明确物理层、数据链路层、应用层各自负责什么，以及 SPI 这类接口在分层中的位置。

在这个项目里，通信相关设计不只是在“怎么发字节”，还包括：

- 用什么电气形式传输
- 用什么帧格式组织数据
- 电机数据字段如何定义
- 多个电机之间如何统一管理
- 控制参数（如 PID）如何在系统中归属

因此，适合按通信层级来理解和扩展。

---

## 2. 总体分层

可将整个通信过程分为三层：

```text
应用层
  ↓
数据链路层
  ↓
物理层
```

在本项目中的理解分别是：

- **物理层**：解决“信号怎么在线上走”
- **数据链路层**：解决“字节如何组织成一帧并可靠传输”
- **应用层**：解决“这一帧数据对电机控制而言代表什么”

---

## 3. 物理层

## 3.1 物理层关注的问题

物理层主要关心：

- 电气形式
- 布线方式
- 通信拓扑
- 时钟方式
- 收发方向
- 传输距离
- 传输频率

它回答的是：**数据到底通过什么硬件方式发送和接收。**

## 3.2 电气形式

### 单端

单端通信通常是一根信号线加一根 GND，通过对地电压判断高低电平。

特点：

- 结构简单
- 成本较低
- 适合短距离、板内通信
- 抗干扰能力相对较弱

### 差分

差分通信使用两根信号线，通过两根线之间的电压差来表示数据。

特点：

- 抗干扰能力强
- 更适合长距离通信
- 常配合双绞线使用
- 更适合工业环境或复杂电磁环境

## 3.3 拓扑结构

常见拓扑包括：

### 一主多从

- 一个主设备统一发起通信
- 多个从设备响应主设备
- 常见于 SPI 等主从结构

#### 主从关系的含义

主从关系指的是：**谁主动发起通信，谁被动响应通信。**

其中：

- **主设备（Master）**：决定什么时候开始通信、和谁通信，很多协议里还负责提供时钟
- **从设备（Slave）**：通常等待主设备访问，在被选中后进行响应

也就是说，通信节奏通常由主设备控制，从设备一般不能随意主动发起完整通信。

#### 在不同通信方式中的表现

- **SPI**：典型一主多从。主设备通过 `CS/NSS` 选择从设备，并通过时钟驱动通信
- **I2C**：常见单主多从。主设备通过地址访问从设备
- **UART / RS232**：通常不严格强调主从，更常见是点对点收发
- **RS485**：是否表现为主从，通常取决于上层协议设计
- **CAN**：通常是多主，不属于典型主从结构

#### 为什么要区分主从

理解主从关系，有助于判断：

- 谁控制总线时序
- 谁负责发起请求
- 从设备是否可以主动上报
- 软件设计里谁负责轮询、调度、地址选择或片选控制

### 点对点

- 两个设备直接通信
- 结构简单
- 调试容易

### 总线型

- 多个节点共享一条总线
- 可以是一主多从，也可以是多主
- 适合多设备挂载场景

## 3.4 时钟方式

### 同步通信

- 有独立时钟线
- 收发双方依赖统一时钟
- 典型例子：SPI

### 异步通信

- 没有独立时钟线
- 双方依靠约定波特率进行收发
- 典型例子：UART、部分串口协议

### 自同步

- 没有单独时钟线
- 但协议本身可以从信号中恢复同步信息
- 常见于更底层总线编码机制

## 3.5 收发方向

### 全双工

- 发送和接收可同时进行

### 半双工

- 发送和接收不能同时进行
- 同一时刻只能做一个方向

## 3.6 常见通信方式

下面列出开发中常见的几类通信方式，方便从分层角度快速理解，也便于后续做接口选型。

### UART / 串口

#### 典型特征

- 单端
- 异步
- 通常为全双工（TX/RX 分离）
- 通常点对点
- 硬件简单，实现成本低

#### 工作方式

UART 本质上是一种异步串行通信方式。双方不共享独立时钟线，而是提前约定：

- 波特率
- 数据位长度
- 校验位
- 停止位

发送端按约定格式一位一位发出，接收端按同样时序采样。

#### 优点

- 接线简单
- 软件和硬件都容易实现
- 调试工具丰富
- 很适合做日志、调试口、简单外设通信

#### 局限

- 抗干扰能力一般
- 长距离性能有限
- 本身不解决多设备共享总线的问题
- 只有“字节流”概念，没有天然强约束的报文边界

#### 适用场景

- 上位机与下位机通信
- MCU 调试输出
- 传感器或执行器串口协议
- 简单控制器之间的点对点通信

#### 分层理解

- 物理层：TX / RX 电平与连线
- 链路层：起始位、停止位、波特率，以及上层自定义帧结构
- 应用层：命令字、状态字、控制参数等业务语义

#### 在项目中的启发

如果你的设备底层表现为“连续字节流”，那它通常更接近 UART/串口思路。这种情况下，软件里往往要额外处理：

- 缓冲区累计
- 找帧头
- 判断长度
- 校验 CRC 或 checksum
- 拆出完整报文

这也是为什么串口项目里，链路层代码通常会比较重。

### RS232

#### 典型特征

- 单端
- 异步
- 常见于点对点
- 常用于设备间串口通信
- 属于较传统的串行标准

#### 工作方式

RS232 可以理解为“串口通信的一种经典电气标准”。

它重点规定的是：

- 电压范围
- 信号含义
- 接口连接方式

也就是说，UART 更像通信方式，RS232 更像这种方式所使用的一套具体电气规范。

#### 优点

- 标准成熟
- 工业和老设备中仍然常见
- 点对点通信实现直接

#### 局限

- 单端传输，抗干扰能力有限
- 不适合现代高密度、多节点总线型场景
- 在嵌入式板内通信中已经不如 TTL 串口和 RS485 常见

#### 适用场景

- 工控设备
- 老设备联调
- PC 与外设连接
- 一些调试和维护接口

#### 分层理解

- RS232 主要定义的是物理层电气规范
- 真正的帧格式和命令语义，仍要靠上层协议决定

### RS485

#### 典型特征

- 差分
- 常见为异步
- 半双工常见
- 适合总线型多节点
- 抗干扰强，适合较长距离

#### 工作方式

RS485 也是一种电气标准，但相比 RS232 更偏向工业现场使用。

它常见的工作形式是：

- 一对差分线挂多个节点
- 总线上同一时刻通常只有一个节点发送
- 其他节点接收或等待轮到自己响应

#### 优点

- 抗共模干扰能力强
- 传输距离较长
- 适合多从站挂接
- 很适合工业设备联网

#### 局限

- 总线访问需要协议协调
- 常见半双工，收发不能随意同时进行
- 设备多时要考虑终端匹配、偏置电阻、地址分配等问题

#### 适用场景

- 工业现场设备联网
- 电机驱动器、传感器总线
- PLC 与从站设备通信
- 多设备远距离布线

#### 分层理解

- RS485 主要定义差分物理层
- 上层常叠加 Modbus RTU 或自定义协议
- 如果做项目实现，RS485 驱动层负责方向控制和收发，上层再负责组帧与解析

#### 在电机场景中的意义

很多电机或驱动器虽然“说自己是 485 通信”，但实际只说明底层电气接口是 RS485。真正决定你怎么发命令、怎么读状态的，仍然是它上面的私有协议或标准协议。

### SPI

#### 典型特征

- 单端
- 同步
- 全双工
- 主从式
- 速度高，适合板级短距离通信

#### 工作方式

SPI 一般包含以下几类信号线：

- SCLK：时钟
- MOSI：主发从收
- MISO：主收从发
- CS / NSS：片选

主设备通过时钟驱动数据收发，并通过片选决定当前与哪个从设备通信。

#### 优点

- 速率高
- 时序直接
- 实现简单
- 适合板内短距离高速外设
- 可同时收发数据

#### 局限

- 一般不适合长距离
- 多从设备时需要更多片选线
- 没有统一的上层报文标准
- 不同设备对时钟极性、相位、字长要求可能不同

#### 适用场景

- MCU 与传感器、编码器、Flash 芯片通信
- 板内高速外设连接
- 对实时性较高的短距离外设交互

#### 分层理解

- SPI 本身主要属于物理层接口
- 上层仍需自行定义帧格式和业务含义
- 如果接入电机控制，往往还需要补一层 SPI 上层协议

#### 在项目中的启发

如果后续要做 `SPIInterface`，它最适合只负责：

- 打开 SPI 设备
- 配置 mode / bits per word / max speed
- 控制片选
- 收发原始字节

而不要把电机协议语义直接塞进硬件接口类里。

### I2C

#### 典型特征

- 单端
- 同步
- 半双工特征更明显
- 多主多从都可实现，但常见为单主多从
- 两根信号线即可完成通信

#### 工作方式

I2C 常见信号线为：

- SCL：时钟线
- SDA：数据线

总线上每个从设备通常有自己的地址，主机通过地址选择要访问的设备。

#### 优点

- 接线少
- 多设备挂接方便
- 自带地址机制和应答机制
- 很适合板内低速配置类和状态类外设

#### 局限

- 速度通常低于 SPI
- 总线电容和上拉设计会影响稳定性
- 不适合高吞吐、大实时性的场景
- 多设备挂接时地址冲突需要处理

#### 适用场景

- 板内低速外设连接
- 传感器
- EEPROM
- IO 扩展器
- 配置类芯片

#### 分层理解

- I2C 介于“物理层接口”和“带部分链路属性的总线”之间
- 地址应答机制是它比普通串口更强的一点
- 但具体寄存器意义、控制命令、状态定义仍属于应用层

### CAN

#### 典型特征

- 差分
- 自同步
- 半双工总线特性明显
- 多主总线
- 抗干扰强，实时性好

#### 工作方式

CAN 的特点是多个节点共享总线，并通过仲裁机制决定谁先发送。它不是简单的“主发从收”，而是允许多个节点主动发消息。

在很多控制系统里，更常见的是：

- 每个设备按 ID 发送或接收特定帧
- 总线上天然是报文通信，而不是纯字节流
- 应用层围绕帧 ID 和数据字段来组织协议

#### 优点

- 实时性好
- 抗干扰能力强
- 非常适合多节点控制系统
- 总线化程度高，布线相对规整

#### 局限

- 报文长度有限
- 调试门槛高于普通串口
- 更适合控制数据，不适合大数据量传输
- 需要合理规划 ID 与总线负载

#### 适用场景

- 汽车电子
- 机器人多电机控制
- 工业控制网络
- 分布式执行器和传感器系统

#### 分层理解

- CAN 不只是物理传输，还天然带有较强的链路层属性
- 工程实现中通常仍把 `CANInterface` 放在硬件接口层
- 真正的“某个帧 ID 表示什么、8 字节里每一位是什么意义”，仍然属于上层协议语义

#### 在项目中的意义

本项目当前很多控制逻辑之所以适合走 CAN，是因为 CAN 很适合：

- 多电机并挂
- 周期性反馈
- 指令广播或分组发送
- 对实时控制较敏感的场景

### Ethernet / 以太网

#### 典型特征

- 常见为差分传输
- 高速
- 支持较远距离和网络化通信
- 支持交换机、路由等复杂网络拓扑

#### 工作方式

以太网本身更像一个更大的网络通信基础。它通常不是只解决“两个芯片怎么连”，而是进一步支持：

- 局域网通信
- 多主机互联
- 上层网络协议叠加
- 分布式系统通信

#### 优点

- 带宽高
- 网络化能力强
- 适合远程访问和跨设备协同
- 工具链成熟

#### 局限

- 协议栈更复杂
- 实现成本和系统复杂度高于 UART / SPI / I2C
- 对强实时闭环控制不一定最优

#### 适用场景

- 上位机与控制器通信
- 分布式设备联网
- 视觉数据传输
- 日志、参数配置、远程控制
- 多机系统协同

#### 分层理解

- 以太网更多是完整网络体系的入口
- 上面常继续叠加 TCP/IP、UDP、HTTP、ROS2 DDS 等协议
- 在机器人系统里，它更常承担“系统级联网”角色，而不是最底层的单个外设直连角色

### 选型建议

- **单个板内外设，追求高速**：优先考虑 SPI
- **多个板内低速器件**：优先考虑 I2C
- **简单点对点调试或单设备通信**：优先考虑 UART
- **工业现场、多节点、长距离**：优先考虑 RS485
- **多电机并挂、实时性和抗干扰都重要**：优先考虑 CAN
- **上位机联网、远程控制、视觉或大数据传输**：优先考虑 Ethernet

## 3.8 物理层在项目中的对应

在本项目里，物理层对应的是各种具体硬件接口，例如：

- `CANInterface`
- `SerialInterface`
- 后续如果引入 SPI，也应有类似的 `SPIInterface`

这一层的职责是：

- 打开设备
- 配置波特率/时序等硬件参数
- 完成底层收发
- 向上提供统一的数据发送/接收能力

也就是说，这一层不关心“某个字节表示转速还是位置”，只关心“字节能不能正确发出去、收回来”。

## 4. 数据链路层

## 4.1 数据链路层关注的问题

数据链路层是“物理链路 + 链路协议”。

它主要解决：

- 一帧数据怎么组织
- 帧头、帧尾怎么定义
- 长度字段怎么放
- 校验如何计算
- 一个收发周期如何进行
- 如何从字节流中切出完整报文

所以它回答的是：**原始字节如何构成可识别、可解析的通信帧。**

## 4.2 链路协议的本质

链路协议就是传递帧格式的规则。

这些规则既可能由硬件直接实现，也可能由软件实现。

例如：

- 有些总线本身已经规定了帧结构
- 有些串口/自定义接口只是负责传字节，真正的帧格式需要软件自己封装和解析

## 4.3 本项目中的理解

在当前项目语境下，数据链路层主要体现在：

- 帧的打包与解包
- 字节流累计接收
- 完整报文提取
- 校验与合法性判断

例如串口通信中，底层接口收上来的是连续字节流，而不是天然分包的数据帧。因此需要链路层逻辑去做：

- 累积接收
- 找帧头
- 判断长度
- 校验 CRC / checksum
- 提取完整包

这类职责在实现上通常会落在接口类和协议类之间的边界位置。

## 4.4 can_interface 的位置理解

你提到“代码为 `can_interface`”，可以理解为它承担了较靠近链路层的一部分职责。

原因是 CAN 这类接口通常不仅涉及底层发收，还天然带有帧化的数据组织方式，因此它比“纯字节流串口”更接近一个已经定义好的链路层载体。

但从工程实现角度看，`CANInterface` 仍然常被归入“硬件接口层”，因为它直接管理设备与收发。更准确地说：

- **从通信原理看**：它横跨物理层和部分链路层
- **从代码分层看**：它通常被放在硬件接口层

## 5. 应用层

## 5.1 应用层关注的问题

应用层关心的是：

- 一帧数据里的字段分别表示什么
- 某种电机协议如何编码与解码
- 多个电机如何管理
- 控制命令如何映射到协议字段
- 反馈数据如何还原为位置、速度、电流、温度等状态
- PID、模式切换、状态机等逻辑如何组织

它回答的是：**这帧数据在电机控制语义上是什么意思。**

## 5.2 在本项目中的对应

本项目的应用层可分成两类：

### 1）协议解释层

例如：

- `UnitreeMotorNative`
- `DJIMotor`
- `DamiaoMotor`

这些模块负责：

- 按具体电机协议编码命令
- 解析电机返回数据
- 定义字段和物理量之间的映射关系

它们不一定直接管理底层硬件，而是专注于“协议语义”。

### 2）系统管理与控制层

例如：

- `motor_control_node`

这一层负责：

- 管理多个电机
- 协调多个硬件接口
- 分发控制命令
- 汇总反馈状态
- 挂接配置参数
- 管理 PID 等控制逻辑

因此，从工程视角看，应用层不只是“协议字段解释”，还包括“围绕电机业务建立起来的控制与管理逻辑”。

---

## 6. SPI 在分层中的位置

## 6.1 SPI 首先属于物理层接口

SPI 的典型特征是：

- 单端
- 同步
- 全双工
- 主从式

所以从通信分层上看，**SPI 首先是一种物理层/硬件接口形式**。

它定义了：

- 时钟怎么走
- 数据线怎么接
- 主从关系如何建立
- 每个比特如何在时钟边沿上传输

## 6.2 SPI 本身不等于完整应用协议

SPI 只解决“怎么传输比特和字节”，并不天然定义：

- 某个字节表示什么含义
- 一帧多长
- 怎么做校验
- 某个命令对应什么电机动作

因此，如果项目要支持 SPI 控制电机，通常还需要在 SPI 之上再叠加：

- 链路层帧格式
- 电机应用协议

也就是说：

```text
电机控制语义
  ↓
SPI 上层协议/帧格式
  ↓
SPI 硬件接口
```

## 6.3 在本项目中的建议落点

如果后续要在本项目中加入 SPI，可以这样理解分工：

### `SPIInterface`

负责：

- 设备打开与关闭
- SPI 模式配置
- 时钟频率配置
- 片选控制
- 原始字节收发

属于：**硬件层 / 物理层接口实现**

### SPI 链路协议处理

负责：

- 定义一帧的结构
- 打包命令帧
- 拆包反馈帧
- 校验与边界处理

属于：**数据链路层**

### 电机协议模块

负责：

- 将目标位置/速度/力矩编码为协议字段
- 将反馈字节解析为状态量
- 管理电机协议语义

属于：**应用层**

## 7. 结合本项目代码的分层映射

可以把当前工程中的通信相关模块粗略映射为：

```text
应用层
  - motor_control_node
  - UnitreeMotorNative
  - DJIMotor
  - DamiaoMotor

数据链路层
  - 各类帧打包/解包逻辑
  - send/recv 后的报文拼接与解析
  - 校验、长度、帧边界处理

物理层
  - CANInterface
  - SerialInterface
  - （未来可扩展）SPIInterface
```

其中需要注意：

- 某些类在代码实现上可能同时跨两层
- 这是工程中常见现象，不必强行绝对割裂
- 关键是明确“每段逻辑主要解决的问题是什么”

这样在后续扩展接口时，才能避免把“硬件收发”“帧协议”“电机语义”混在一起。

---

## 8. 分层带来的好处

按这种方式理解通信层级，有几个直接好处：

### 1）更容易扩展新接口

比如新增 SPI 时，只要补充：

- `SPIInterface`
- SPI 对应的链路层处理
- 必要的协议适配

而应用层整体设计可以尽量保持不变。

### 2）更容易定位问题

出现通信异常时，可以先判断问题位于哪一层：

- 没有数据：先看物理层
- 收到乱码或组包失败：看链路层
- 数据能收发但电机动作不对：看应用层

### 3）更容易统一 CAN / 串口 / SPI

虽然不同接口的底层特性不同，但只要分层清晰，就能把共同逻辑抽出来：

- 硬件接口负责收发
- 协议模块负责编解码
- 控制节点负责调度和管理

这样系统结构会更稳定，也更便于维护。

---

## 9. 总结

本项目中的通信可以按以下思路理解：

- **物理层**：解决“怎么传”
  - 单端/差分、同步/异步、全双工/半双工、拓扑、距离、频率
  - 对应 `CANInterface`、`SerialInterface`、未来的 `SPIInterface`

- **数据链路层**：解决“怎么成帧”
  - 帧格式、长度、校验、组包、拆包、字节流到报文的转换

- **应用层**：解决“这帧数据是什么意思”
  - 电机协议语义、状态解析、多电机管理、PID 与控制逻辑

对于 SPI，应把它看成：

- 底层是一种**同步主从硬件接口**
- 上面仍然需要补充**链路协议**
- 再往上才是**电机控制应用语义**

这个分层思路适合用于后续对通信接口进行修改、扩展和统一。
